# Decision Log

> Required. Record semantic decisions before modifying billing, usage, routing, failover, release, or customer-facing contract behavior.

## Business glossary

| Term | Meaning in this task | Source of truth | Must not be confused with |
|---|---|---|---|
| balance | In billing context, customer wallet balance derived from append-only ledger entries. In release context, not a money term. | Customer billing ledger for money; ALB / rollout state for traffic weights. | Provider account balance / load-balancing weight / customer wallet |
| account | Customer account is the tenant/team being billed; provider account is the upstream vendor account; upstream credential is the auth identity used to call a provider. | Customer DB for tenant identity; provider config/auth pool for upstream credentials. | Customer account / provider account / upstream credential |
| usage | Raw metered activity generated by gateway or batch processing; separate from payable ledger entry and dashboard aggregate. | Raw usage metering records and official usage reporting pipeline. | Raw usage event / ledger entry / dashboard aggregate |
| total cost | Customer-facing wording is ambiguous. For this incident, `$100.00` means official list-price usage, not wallet debit. | Official usage reporting plus list-price pricing rules. | Official list price / customer payable amount |
| actual cost | Customer payable amount after prepaid multiplier. For Acme example, `$40.00`. | Billing ledger debit entry after applying team prepaid multiplier. | Deducted ledger amount / provider settlement amount |
| official cost | List-price usage value used for reporting and finance usage visibility. | Raw usage metering plus official pricing rules. | Customer payable debit |
| credit | Append-only ledger event that increases customer wallet balance or reverses a prior debit. | Billing ledger. | Provider credit, promotional copy, or load weight |
| prepaid | Customer package multiplier used to compute payable debit from official usage. | Customer billing contract / team prepaid package metadata. | Provider discount |
| stable | Current production stable target group/image in rollout state. | `ops/current-rollout-state.json` / release controller. | Git branch / ALB stable target group / stable API contract |
| canary | Public canary target group/image currently receiving 1% real traffic. | `ops/current-rollout-state.json` / ALB weights. | Private shadow canary / public traffic canary |

## Decision entries

### 2026-06-14 19:14 — Billing display semantics

- Context: Acme Team sees `Total usage cost: $100.00` while prepaid wallet debit is `$40.00` with multiplier `0.4`.
- Decision: classify the incident as dashboard wording / aggregate semantic mismatch, not as proven wrong debit, undercharge, or double billing.
- Source of truth: official usage reporting preserves `$100.00`; customer balance source of truth is the billing ledger preserving `$40.00` payable debit.
- Alternatives rejected: rewriting raw usage to `$40.00`, rewriting historical ledger entries, adding dashboard-only billing math, or treating current evidence as double billing.
- Risk: customer-facing language can still create distrust if the dashboard does not distinguish list-price usage from wallet debit.
- Verification: tests or smoke checks must prove official usage remains `$100.00`, payable debit remains `$40.00`, provider balance is untouched, and load-balancing weights are untouched.
